Chinchilla-optimal floor vs. production-standard cost

The theoretical compute floor for a given model size (Chinchilla-optimal, ~20 tokens/param) versus what a production run realistically costs. Production runs cost more than the floor for two documented reasons, shown separately below. This does not claim a lower cost for the same model — a given size costs the same 6·N·D FLOPs for everyone. It shows that a lean operation can run near the floor, while production standards carry a defined premium.
How the bands are built (all assumptions stated, none measured except our own runs):
1 · Over-training band (token axis). Chinchilla-optimal is ~20 tokens/param. Production models deliberately train far past that to make inference cheaper — Llama 3 8B used ~15T tokens (~1,900 tokens/param). We model production token budgets as 5×–100× the Chinchilla-optimal count; more tokens = more compute = higher cost.
2 · Operational overhead band (cost multiplier). On top of raw compute: failed runs, redundant checkpointing, restart waste (exactly the kind we hit on the 1.5B run, scaled up), and safety margin. Modeled as 1.5×–3× over compute cost. This is an assumption, not a measurement.
3 · Our measured runs. Plotted at their actual token ratio and measured cost. Note honestly: our current budgets are below Chinchilla-optimal (e.g. 1.5B on 15B tokens ≈ 10 tokens/param), so these are demonstration runs — cheaper, but intentionally under-trained relative to a production model of the same size.
Bottom line for the reader: the gap between our points and the production band is real, but part of it is "we run leaner" and part of it is "we train smaller budgets." Both are stated so the comparison is honest.
